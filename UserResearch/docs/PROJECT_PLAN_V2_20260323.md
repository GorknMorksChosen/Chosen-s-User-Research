# 项目计划 V2（任务路线图版）

> 更新日期：2026-03-23  
> 计划定位：个人技术代表作（Masterpiece）演进路线图。  
> 核心原则：绝对的统计严谨性、深度的现代 Python 工程化、极具说服力的全链路 Demo。

---

## 一、路线图概览

- **P0 统计底盘与架构收口**：消灭统计错配，实现 Core/Web 物理隔离。
- **P1 现代工程化武装**：强类型契约、混沌 Mock 测试引擎、流式内存治理。
- **P2 高阶洞察与自动化展示**：带版本元数据的跨期基线库、自动生成多级看板报告。

---

## 二、优先级任务清单

### P0-A 统计口径一致性（最高优先级）

**目标**：保证“同一输入 -> 同一结论 -> 同一导出”。

- 任务 A1：修正矩阵评分与分支路由一致性
- 任务 A2：统一小样本策略（参数/非参数切换规则）
- 任务 A3：统一多选提及编码规则（`0` / `0.0` / 空文本 / 缺失）
- 任务 A4：统一显著性阈值 `alpha` 在引擎、UI、导出三层口径

**验收方式（DoD）**：
- 用同一测试数据分别跑 Web/CLI/Pipeline，核心统计结论一致
- 新增关键分支自动化测试（单选/多选/评分/矩阵评分）全部通过
- 小样本/空样本场景有明确输出与用户提示，不出现静默失败

**证据产物**：
- 变更前后对照表（至少 3 份黄金样例）
- 测试报告（含失败分支已修复说明）

---

### P0-B 架构边界收口（core/web 单一事实源）

**目标**：核心计算只在 `survey_tools/core/`，Web 仅负责交互展示。

- 任务 B1：收口重复实现（同类分析逻辑不再双维护）
- 任务 B2：移除 core 层对 UI 依赖（例如 streamlit 直接调用）
- 任务 B3：Web 层改为调用 core 服务函数，不保留统计实现细节

**验收方式（DoD）**：
- core 层无 UI 框架依赖
- 同类分析逻辑有唯一入口函数（有映射清单）
- 回归测试通过，历史关键功能不回退

**证据产物**：
- “旧实现 -> 新入口”映射表
- 迁移影响面清单（文件与函数级）

---

### P0-C 文档与入口口径统一

**目标**：README、注册表、启动菜单、CLI/Web 能力描述完全一致。

- 任务 C1：统一工具数量与分类口径
- 任务 C2：统一各工具能力描述（尤其 CLI 与 Web 差异）
- 任务 C3：补齐环境模板与使用说明（含可选能力边界）

**验收方式（DoD）**：
- README 与 `tool_registry.py` 信息逐项核对无冲突
- 用户按 README 操作可直达对应能力，不出现“文档写了但入口不可达”
- 重要说明在 `README.md` 与 `docs/DEV_LOG.md` 有可追踪记录

**证据产物**：
- 文档对齐核对表（建议 1 页）

---

### P1-A 测试体系升级（保留旧脚本并行）

**目标**：建立可持续质量门禁，不依赖人工回忆。

- 任务 D1：建立最小 `pytest` 目录与基线套件
- 任务 D2：迁移关键统计分支用例（参数化覆盖边界）
- 任务 D3：保留既有验证脚本并行两周，逐步替换

**验收方式（DoD）**：
- CI 必跑最小套件（冒烟 + 统计关键分支）
- 关键回归失败会阻断合并（红灯机制生效）
- 每周输出一次失败原因与修复闭环记录

---

### P1-B 深度数据契约与强类型重构

**目标**：消除字典乱飞与隐式调用，打造高安全感的现代 Python 代码库。

- 任务 E1：全面引入 Pydantic V2（优先覆盖 `QuestionSpec`、分群配置、前端传参；按模块分批落地）。
- 任务 E2：用 `typing` 为 Core 层函数补齐类型提示，推进静态检查（MyPy/Pyright）持续收敛。
- 任务 E3：依赖闭环（`requirements` / lock / matrix 一致）与自动化环境校验。

**验收方式（DoD）**：
- Core 层静态检查无高优先级类型告警（目标全绿，允许分阶段收敛）。
- 异常输入能在 Pydantic 层快速拦截并输出可读 `ValidationError`。
- 新环境按 lock 文件可稳定复现并通过依赖矩阵检查。

**实施边界（强约束）**：
- 采用“模块分批 + 可回滚”策略，避免一次性大手术影响主线交付。
- 对高风险模块先做适配层，不直接硬改核心统计逻辑。

---

### P1-C 性能与内存治理（新增，P1 必做）

**目标**：保证大样本场景可运行，不因内存峰值导致崩溃。

- 任务 P1C-1：排查高风险内存路径（上传、导出、下载、跨期对比）
- 任务 P1C-2：导出与下载链路改造为流式处理，避免整包读入内存
- 任务 P1C-3：建立大样本性能基线（耗时、峰值内存、失败阈值）

**验收方式（DoD）**：
- 大样本演练可完整跑通，无 OOM
- 关键链路不存在一次性全量字节吞吐模式
- 输出性能对比记录（改造前后耗时/峰值内存）

---

### P1-D 混沌测试与故事化 Mock 引擎（新增）

**目标**：用高真实感模拟数据证明系统“算得对、扛得住、讲得清”。

- 任务 M1：重构 `generate_mock_data.py`，支持按分布（正态/偏态/双峰）生成打分数据。
- 任务 M2：引入混沌注入（直线作答、极端离群、文本噪声、编码异常等）。
- 任务 M3：配置化生成“故事化对比样本”（例如优化前/优化后同类玩家群体）。

**验收方式（DoD）**：
- 引擎可稳定生成十万级样本用于压测。
- 注入脏数据后系统能优雅降级并输出告警，不出现底层崩溃。
- 至少 2 套故事化样本被用于 P2 阶段 Demo 报告。

---

### P2-A 高阶业务放大器：带上下文的基线库与复盘

**目标**：从“冷指标对比”升级为“带版本语境的可解释洞察”。

- 任务 F0（前置）：设计支持动态 Schema 演进与**游戏上下文元数据**的存储模型（必须含：版本号、测试节点、核心改动说明）。
- 任务 F1：在 F0 通过后建立轻量基线库（SQLite 优先）
- 任务 F2：结合 M3 的故事化样本与真实数据，跑通自动沉淀与对比流程（按指标字典版本化）
- 任务 F3：实现“本次 vs 上次 vs 历史基线”三段式对比

**验收方式（DoD）**：
- F0 评审通过（至少覆盖 3 个不同问卷版本的兼容方案与元数据映射）。
- 至少 2~3 次历史数据可自动对比
- 输出包含变化量与显著性（口径统一）
- 对比结果可关联业务动作（如版本改动后满意度显著提升/下降）

---

### P2-B 业务放大器：管理层可读输出与文本全链路

**目标**：把分析结果转成可执行决策。

- 任务 G1：老板视角看板（仅保留“3 个关键结论”）
- 任务 G2：一键 3 页汇报稿（Markdown/PDF）
- 任务 G3：文本领域词典与 few-shot 模板（游戏语境）
- 任务 G4：文本分析接入 Pipeline（替换 placeholder，支持失败降级）

**验收方式（DoD）**：
- 报告阅读时间 <= 3 分钟可获取核心结论
- 每次输出包含：拖累项、分群差异、优先动作
- 业务方反馈可直接用于排期会讨论
- 一键流程可同时产出定量 + 文本结果（文本失败不阻断主流程）

## 三、统一验收框架（所有任务适用）

- **功能验收**：功能可达、结果可解释、异常可提示
- **一致性验收**：Web/CLI/Pipeline 结论一致
- **回归验收**：黄金样例对比 + 自动化测试通过
- **文档验收**：README 与 DEV_LOG 同步留痕
- **可维护性验收**：入口唯一、依赖清晰、责任边界明确

## 四、风险预案与约束

### 高风险与预案

- **风险 1：修复后结果变化引发业务质疑**  
  预案：保留黄金样例快照，发布“口径变更说明 + 前后对照”。

- **风险 2：重构导致回归**  
  预案：先迁移后删除，分批合并，每批必须通过冒烟与关键测试。

- **风险 3：测试迁移拖慢迭代**  
  预案：旧脚本并行，按优先级迁移，不做一次性替换。

### 执行约束

- 未通过 P0-A 前，不新增复杂业务功能
- 新增能力必须附带最小回归用例
- 任何口径变化必须同步 README 与 DEV_LOG
- 未完成 P1-C 前，不上线新增重型分析功能

---

## 五、执行机制（Craftsmanship）

作为技术代表作，开发节奏不以时间盒为硬约束，而以“架构质量 + 代码质量 + 可演示性”为主评价标准。

- **每次提交的自检要求**：
  - 新增核心函数是否有清晰 Docstring 与类型签名；
  - 是否有对应 Pytest 覆盖主逻辑与边界情况；
  - 是否补充了必要的回滚说明或降级策略。

- **阶段复盘只看三件事**：
  - 静态代码质量是否提升（类型告警是否持续下降）；
  - 系统对脏数据和大样本的抵抗力是否增强；
  - 核心逻辑抽象是否更内聚（能否跨入口复用而不复制实现）。
