# AI Review Prompt（项目审查模板）

请你作为本项目的高级代码审查员，对仓库进行全量 review，并严格围绕以下四个维度输出结论。  
审查时请以项目规则文件 `.cursorrules` 为最高约束之一，并结合 `README.md`、`docs/DEV_LOG.md`、`survey_tools/core/`、`survey_tools/web/` 的实际实现给出证据。

---

## 1) 统计严密性（必须重点检查）

请逐项判断并给出“符合 / 部分符合 / 不符合”：

- 是否遵循 `.cursorrules` 要求：先检查正态性与方差齐性，再决定参数/非参数检验路径？
- 当不满足适用条件时，是否明确回退到非参数或稳健方法？是否在结果中注明原因？
- `p-value` 的计算链路是否严谨（包含多重比较场景下的校正策略）？
- 效应量（Effect Size）计算是否与检验方法匹配，且数值口径在 UI、导出、日志中一致？

请重点关注函数/模块（可扩展）：
- `survey_tools/core/quant.py`
- `survey_tools/core/advanced_modeling.py`
- `survey_tools/core/effect_size.py`

---

## 2) 架构解耦（web/core 边界）

请判断是否存在 UI 层与算法层逻辑渗透，重点检查：

- `survey_tools/web/` 中是否出现统计计算、数据清洗、口径决策等“应在 core 的逻辑”？
- `survey_tools/core/` 是否反向依赖 Streamlit 或 UI 状态？
- 是否存在“web 与 core 双轨重复实现”导致口径漂移风险？
- 是否存在同一分析能力多个入口实现不一致（Web/CLI/Pipeline）的情况？

请给出可执行建议：哪些逻辑应收敛到 `survey_tools/core/`，哪些仅保留在 `survey_tools/web/`。

---

## 3) 异常流与健壮性（边界输入）

请重点审查以下异常输入场景：

- 用户上传全空列数据；
- 某些列全为相同值（例如直线作答）；
- 文件中含特殊字符、异常编码、损坏/异常 Excel 结构；
- 小样本、缺失率过高、分组列无效等场景。

请判断系统行为是：
- 崩溃（抛异常中断）；
- 静默失败（吞错继续）；
- 或优雅报错（给出可读提示并安全回退）。

请明确指出：
- 哪些位置使用了 `except Exception` 且可能掩盖问题；
- 哪些模块具备明确的错误提示与降级路径；
- 哪些地方需要补充输入校验与结构化告警。

---

## 4) Streamlit 特性（状态管理与资源风险）

请重点检查 `st.session_state` 的使用是否存在：

- 大对象长期驻留导致的内存风险（例如大字节导出结果、全量 DataFrame 缓存）；
- 多页签/多流程切换导致的状态污染；
- 文件切换后状态未清理，造成“旧结果污染新数据”；
- 按钮触发 rerun 后状态不一致（例如导出按钮失效、流程中断）。

请给出“可落地”的状态治理建议（键命名规范、生命周期清理、按需缓存策略）。

---

## 输出要求（必须遵守）

1. 按严重程度输出问题：`P0 > P1 > P2`。  
2. 每个问题都要包含：`风险描述`、`证据（文件/函数）`、`影响`、`修复建议`。  
3. 最后输出：
   - “是否符合 `.cursorrules` 统计标准”的总评；
   - “下一步 To Do（按优先级）”；
   - “建议新增的最小测试集”。
4. 若未发现问题，请明确写“未发现阻断问题”，并说明残余风险与测试盲区。
